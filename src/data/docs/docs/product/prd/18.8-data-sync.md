## Purpose

Provide encrypted local storage and multi-backend sync (iCloud CloudKit + Supabase PostgreSQL) with zero-knowledge architecture ÔÇö ensuring user health data never leaves the device unencrypted.

## Business Objective

Privacy-first architecture is FitMe's primary differentiator. Zero-knowledge encryption and multi-backend sync enable the "privacy-first fitness" positioning that no competitor matches. Double-layer encryption with HMAC integrity is security beyond industry standard.

## Target Persona(s)

- **All personas** ÔÇö data storage is infrastructure
- **Primary differentiator for:** The Data-Driven Optimizer (cares deeply about data privacy)

## Current Implementation

### Double-Layer Encryption
| Layer | Technology | Purpose |
|-------|------------|---------|
| Layer 1 | AES-256-GCM (CryptoKit) | Primary encryption of plaintext |
| Layer 2 | ChaCha20-Poly1305 (CryptoKit) | Secondary encryption of AES ciphertext |
| Integrity | HMAC-SHA512 | Per-record tampering detection |
| Key Storage | Keychain with biometric ACL | `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` |
| Key Derivation | HKDF-SHA512 | For key rotation |
| File Protection | iOS `.completeFileProtectionUnlessOpen` | OS-level encryption while locked |

### Container Format
```
[version byte][timestamp 8B][HMAC 64B][ChaCha20 ciphertext]
```

### Sync Backends

#### CloudKit (Apple-only)
- Private Database per user (RLS via iCloud account)
- **Sync triggers:** app active Ôćĺ fetchChanges + subscribeRealtime; app background Ôćĺ unsubscribeRealtime + pushPendingChanges; first login Ôćĺ fetchAllRecords
- **Push:** Encrypt Ôćĺ upsert to `sync_records` (per-record error isolation)
- **Pull:** Query by record type, paginated (~200 records/page)
- **Cardio images:** Encrypt Ôćĺ CKAsset field; metadata in `cardio_assets` table

#### Supabase (Cross-platform)
- PostgreSQL + Realtime subscriptions + Storage buckets
- Tables: `sync_records`, `cardio_assets`
- RLS: `user_id` matches session
- **Push:** Encrypt Ôćĺ base64 Ôćĺ upsert with SHA-256 checksum
- **Pull:** Query by `last_modified >= lastPull`, decrypt each row
- **Realtime:** Subscribe to INSERT/UPDATE; triggers incremental fetchChanges (re-pulls, doesn't trust payload)
- **Cardio images:** Encrypt Ôćĺ upload to `cardio-images` storage bucket; lazy download

### Conflict Resolution
| Data Type | Strategy |
|-----------|----------|
| DailyLogs | Last-modified-wins by timestamp |
| WeeklySnapshots | Local unsaved changes (needsSync=true) prevent remote overwrite |
| Singletons (Profile, Preferences) | SHA-256 three-way merge: local==remoteÔćĺno-op; local!=lastSyncedÔćĺkeep local; local==lastSyncedÔćĺapply remote |

### EncryptedDataStore
- @MainActor observable ÔÇö in-memory cache of all user data
- Loaded from disk on app launch (after biometric auth)
- Persisted to disk after any modification
- Merge methods: `mergeDailyLog()`, `mergeWeeklySnapshot()`, `mergeProfile()`, `mergePreferences()`, `mergeMealTemplates()`
- `needsSync` flag per record (cleared after successful push)

### EncryptionService
- Actor (thread-safe), shared singleton
- `setSessionContext()` / `clearSessionContext()` for biometric-protected key access
- `encrypt<T: Encodable>()` / `decrypt<T: Decodable>()`
- `rotateKeys()` ÔÇö decrypt all Ôćĺ generate new keys Ôćĺ re-encrypt Ôćĺ persist
