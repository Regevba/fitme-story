## The Starting Point: v1.0-v1.2 (The Monolith)

The original PM workflow was a single script where every work item went through the same 9-phase funnel:

```
Research --> PRD --> Tasks --> UX --> Implement --> Test --> Review --> Merge --> Docs
```

**Problems identified:**
1. Bug fixes blocked behind PRD and UX gates they didn't need
2. No task-level visibility -- dashboard only showed feature-level cards
3. Domain agents existed but weren't wired as parallel workers
4. No cross-feature prioritization
5. No feedback loop -- shipped code wasn't monitored for impact

---

## v2.0: The Hub-and-Spoke Split

**The big shift:** Linear pipeline --> living cycle.

v2.0 replaced the monolith with 11 specialized agents (1 hub + 10 spokes) communicating through a shared data layer of 15 JSON files. It also added Phase 9 (Learn), making the workflow circular -- after shipping, the hub continuously monitors user feedback and dispatches fixes.

```
                   +---------------------+
                   |  /pm-workflow (HUB)  |
                   |  Orchestrates 0-9    |
                   +--+--+--+--+--+------+
                      |  |  |  |  |
              +-------+  |  |  |  +-------+
              v          v  v  v          v
           /research  /ux /design /dev  /release
                        v       v
                      /cx  /marketing  /ops
                        +------+-------+
                               v
                        /pm-workflow (Phase 9: Learn)
```

**Tested on:** Onboarding v2 -- 6 screens, 22 tasks, first feature through the full 10-phase lifecycle.

**Tested on:** Home v2 (723-line rewrite), plus 4 more features shipped in one session.

## v4.0: Reactive Data Mesh and Learning Cache

**Core principle change:** From "data originates in conversations" to "any entry point, any time, data flows."

### Integration Adapter Layer

External data enters through adapters that isolate format changes from consuming agents:

```
Raw MCP response --> schema.json (validate) --> mapping.json (normalize)
    --> validation gate --> shared layer
```

6 adapters shipped: GA4, App Store Connect, Sentry, Firecrawl, Axe, Security Audit.

### Automatic Validation Gate

All incoming data cross-referenced against existing state:
- GREEN (>= 95%): write + notify
- ORANGE (90-95%): write + advisory
- RED (< 90%): block + alert, user resolves

### Learning Cache (L1/L2/L3)

Inspired by CPU cache hierarchy:
- **L1** (per-agent): hot, single-domain patterns
- **L2** (cross-agent): warm, shared by 2+ agents
- **L3** (project-wide): cold, referenced by 5+ agents

**Tested on:** Training v2 -- first feature with cache hits (40%).

---

## v4.1: Agent Internal Lifecycle

Every agent now mirrors the hub internally with a 4-phase lifecycle:

```
Cache Check --> Research (if miss) --> Execute --> Learn
```

**Without the lifecycle:** Agents are stateless functions -- same speed every time.
**With it:** The 4th similar task is ~65% faster than the 1st.

**Tested on:** Nutrition v2 (55% cache), Stats v2 (65% cache), Settings v2 (70% cache).

---

## v4.2: Self-Healing Hub

Phase 0 (Health Check) added to the agent lifecycle. Runs probabilistically (~25% chance, 2h cooldown) with 5 weighted integrity checks:

| Check | Weight |
|-------|--------|
| Cache Staleness | 25% |
| Cache Hit Accuracy | 25% |
| Shared Layer Consistency | 20% |
| Routing Integrity | 15% |
| Adapter Availability | 15% |

**Why it matters:** Without self-verification, a corrupted cache or inconsistent shared layer silently degrades every agent invocation. The health check catches drift before it compounds.

## v5.0-v5.1: SoC-on-Software (The Chip Architecture Phase)

### The Problem

By v4.3, the framework consumed ~96K tokens per session (~48% of the 200K context window) before any work began.

### The Solution: 8 Hardware-Inspired Optimizations

| # | Optimization | Chip Inspiration | Impact |
|---|-------------|-----------------|--------|
| 1 | On-demand agent loading | Apple <Term slug="lora-hot-swap">LoRA hot-swap</Term> | ~30K tokens saved |
| 2 | Cache compression | Apple 3.7-bit <Term slug="palettization">palettization</Term> | ~24K tokens saved |
| 3 | Batch dispatch | <Term slug="tpu">TPU</Term> weight-stationary | 5x fewer dispatches |
| 4 | Result forwarding | UMA zero-copy | Eliminate write-read cycles |
| 5 | Model tiering | ANE mixed-precision | Fast model for mechanical, deep for judgment |
| 6 | <Term slug="speculative-preload">Speculative preload</Term> | Branch prediction | 30-40% latency reduction |
| 7 | Systolic chains | TPU <Term slug="systolic-chain">systolic array</Term> | Eliminate global reads mid-chain |
| 8 | Task complexity gate | ARM <Term slug="big-little">big.LITTLE</Term> | Parallel lightweight + serial heavyweight |

**Combined impact:** ~63% framework overhead reduction (121K --> 45K tokens per phase).

**Tested on:** AI Engine Architecture -- 13 tasks, 17 files, 1.5 hours. 11.3 files/hour (best throughput recorded).

## v5.2: Dispatch Intelligence and Write Safety

Made parallel agent coordination deterministic instead of luck-dependent:

**Sub-Project A: Dispatch Intelligence**
- 3-stage pipeline: score complexity --> probe capability --> dispatch with budget
- Tool budgets per tier (10/25/50) cut average tool usage by 48% and variance by 84%

**Sub-Project B: Parallel Write Safety**
- Snapshot/rollback + 3-tier code region mirror pattern
- Progressive marker learning: each dispatch teaches the system to be faster next time

**Tested on:** 4-feature continuation stress test.

## Key Takeaways

- **Every version was tested on real work.** No version shipped without being validated on an actual feature. The evolution was driven by concrete problems encountered during production use, not theoretical improvements.
- **The monolith-to-microservices transition happened in one day.** v1.2 to v2.0 (monolithic script to 11-agent hub-and-spoke) was a single session. The shared data layer made the transition clean.
- **Learning caches created compounding returns.** The most dramatic improvement came not from architectural changes but from the cache: 0% to 70% hit rate across 6 refactors, producing a 6.5x speedup.
- **Chip architecture principles transfer directly to software frameworks.** On-demand loading, cache compression, batch dispatch, and hybrid task routing aren't metaphors -- they're direct implementations of the same optimization principles, achieving 63% overhead reduction.
- **Measurement closes the loop.** v6.0's deterministic instrumentation of the framework itself completed the final feedback loop: the framework that measures features now measures itself.
